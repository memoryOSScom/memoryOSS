1The Memory Problem

AI agents are stateless by design. Every session starts with zero context from previous interactions. The LLM API receives a prompt, returns a response, and forgets everything.

This creates a concrete, measurable cost:

  • Repeated context — developers re-explain project structure, conventions, and preferences every session. This wastes tokens and time.
  • Lost learnings — an agent that debugged a complex issue yesterday has no memory of the solution today.
  • No cross-session continuity — multi-day projects require manual context transfer (copy-pasting, README files, CLAUDE.md).
  • No cross-provider continuity — knowledge gained while using Claude is invisible when switching to Codex, and vice versa.

The current workarounds — manually maintained context files, conversation exports, project-specific instructions — are fragile, incomplete, and require constant human maintenance. They don't scale.

2Why Existing Approaches Fall Short

Context Files (CLAUDE.md, .cursorrules)

Static instruction files prepended to every request. They work for fixed conventions but can't learn, can't grow, and can't capture the dynamic knowledge an agent accumulates over time. They're written by humans, not agents.

RAG Pipelines (LangChain, LlamaIndex)

Retrieval-Augmented Generation requires building a custom pipeline: chunk documents, embed them, store in a vector database, query at runtime, inject results. This works for document Q&A but is poorly suited for agent memory because:

  • It requires code changes in every application
  • It's designed for static document corpora, not dynamic agent interactions
  • It doesn't handle trust, decay, or fact lifecycle
  • Each application needs its own RAG integration

Cloud Memory Services (Zep, Mem0, etc.)

SaaS platforms that store memories externally. They solve the persistence problem but introduce new ones:

  • Your agent's complete interaction history leaves your infrastructure
  • Latency depends on network conditions
  • Vendor lock-in and pricing uncertainty
  • Still require SDK integration or code changes

What's Missing

Most existing solutions force a tradeoff between transparent integration (zero code changes), local operation (no cloud dependency), cross-provider support (works with any LLM), and memory lifecycle control (trust, decay, and filtering). memoryOSS is designed to combine those properties in one system.

3The Proxy Architecture

memoryOSS takes a pragmatic approach: instead of asking applications to integrate a memory SDK, it positions itself as a proxy between the AI client and the upstream LLM API.

AI Agent (Claude Code, Codex, Cursor, any client)
    |
    | HTTP request (standard API format)
    v
memoryOSS Gateway (:8000)
    |
    |-- 1. Try core   memory-aware path when healthy
    |-- 2. Recall     semantic search across all memory channels
    |-- 3. Inject     prepend relevant memories to system prompt
    |-- 4. Forward    pass enriched request to upstream LLM
    |-- 5. Extract    async: pull facts from LLM response
    |-- 6. Store      embed, score, persist new memories
    |-- 7. Fail open  direct upstream passthrough if core is unavailable
    |
    v
Upstream LLM (Anthropic, OpenAI, any compatible API)

The client sees a standard API endpoint. It sends requests exactly as it would to the real API. In the default setup, a lightweight local gateway sits in front of the memory core: when the core is healthy, memoryOSS intercepts, enriches, forwards, learns, and returns the response; when the core is unavailable, the gateway falls back to direct upstream passthrough so the client keeps working.

Why a Proxy?

The proxy pattern has a critical advantage: near-zero integration friction. OpenAI-compatible clients can be pointed at the local gateway with a base URL override, while the setup wizard configures MCP automatically for Claude Code and Codex. In API-key setups the wizard can also enable the local gateway transparently; in OAuth-first setups it keeps MCP enabled and avoids forcing global BASE_URL overrides so provider login flows continue to work. Codex OAuth therefore stays MCP-first, while proxy mode remains available for API-key clients and other proxy-safe paths. The practical result is a hybrid model: transparent memory on the request path when proxy-safe, explicit memory tools via MCP, and fail-open upstream passthrough if the memory core is unavailable.

# Manual proxy mode (optional, API-key clients)
export ANTHROPIC_BASE_URL=http://127.0.0.1:8000/proxy/anthropic/v1
export OPENAI_BASE_URL=http://127.0.0.1:8000/proxy/v1

This also enables cross-provider memory. Claude and Codex share the same memory store. Knowledge gained with one provider is available to the other. The gateway normalizes across API formats internally — including both the Chat Completions API (/v1/chat/completions) and the Responses API (/v1/responses) — for manual proxy mode. In OAuth-first Codex setups the supported default path remains MCP-first, without forcing global proxy overrides.

MCP Alongside the Gateway

For tools that support the Model Context Protocol (Claude Desktop, Claude Code, Codex), memoryOSS also runs as an MCP server with four tools: store, recall, update, forget. In this mode, the agent explicitly decides when to use memory rather than having it injected transparently. MCP is therefore not the transport failover path; it is the explicit memory-tool path that runs alongside the gateway.

Anthropic Local MCP Submission Path

For Anthropic's local MCP / Claude Desktop extension path, memoryOSS uses the same local stdio server it already ships today: memoryoss -c /absolute/path/to/memoryoss.toml mcp-server. The public package metadata lives in server.json, the operator install path and three end-user examples are documented in the repository README, and the public privacy policy and support contact are published at memoryoss.com/datenschutz.html and hello@memoryoss.com.

The prepared local-MCP examples intentionally match real memory workflows rather than synthetic toy prompts: project continuity recall, preference capture, and deleting an outdated memory. That gives a future directory submission a concrete install path and a practical \"what can I do with this?\" story without inventing a separate packaging-only feature surface.

One packaging gap remains explicit instead of hidden: the current rmcp 0.1.5 transport does not yet emit MCP spec-era tool title, readOnlyHint, and destructiveHint fields on the live tools/list response. memoryOSS therefore documents the exact annotation mapping in code and docs today, but does not claim that a finished Anthropic directory bundle or portable .mcpb artifact already ships in this repository.

Not every request should read and write memory. memoryOSS supports four modes, configurable globally or per-request via HTTP headers:

On a fresh setup the wizard defaults to full. If existing memories are already present, it asks which mode to use and defaults that prompt to full.

4Multi-Channel Retrieval

Most memory and RAG systems rely solely on vector similarity search. This works for semantic meaning ("what's conceptually related?") but fails for exact matches ("what did we decide about the UserService class?"), keyword lookups, and identifier-heavy developer contexts.

memoryOSS combines four retrieval channels, each capturing a different signal:

Vector (semantic) 0.30
BM25 (full-text) 0.30
Exact match 0.25
Recency 0.15

Vector Search (Semantic Similarity)

384-dimensional embeddings via AllMiniLM-L6-V2, indexed with usearch. Finds memories that are conceptually related to the query even when they use different words. Runs locally — no API call for embedding generation.

BM25 Full-Text Search

Powered by tantivy (Rust-native Lucene equivalent). Excels at keyword matching, especially for technical terms, error messages, and code identifiers that don't embed well semantically.

Exact Match

Direct substring matching for identifiers, file paths, function names, and configuration keys. When a developer asks about handleWebSocket, the exact match channel finds memories that mention that specific identifier — something vector search often misses because embeddings compress identifiers into generic semantic space.

Recency Bias

Recent memories get a time-decay boost. A decision made yesterday is more relevant than one from three months ago, all else being equal.

Score Fusion and Deduplication

Channel scores are weighted and combined. Memories that score high in multiple channels rank highest (reinforcement). Duplicate results are merged. The final ranked list goes through MMR (Maximal Marginal Relevance) diversity filtering — preventing the top-N results from all being near-duplicates of each other.

IDF Identifier Boosting

memoryOSS maintains an IDF (Inverse Document Frequency) index across all memories. Rare identifiers — a unique class name, a specific config key — get boosted in retrieval. Common words get dampened. This significantly improves precision in developer-heavy contexts where the query is often a specific technical term.

Precision Gate

A memory must score above a minimum threshold in at least one channel (min_channel_score: 0.15) AND above the global threshold (min_recall_score: 0.40) to be injected. This prevents low-confidence memories from polluting context. The thresholds are currently tuned based on internal testing and may evolve as more usage data becomes available.

5Trust Scoring and Quarantine

Not all memories are equally reliable. A fact extracted from an LLM response might be hallucinated. An old memory might be outdated. A frequently-accessed memory is more likely to be correct (survivorship signal).

memoryOSS implements a 4-signal Bayesian trust scoring system:

  1. Recency Decay — trust decreases with age. Old memories aren't deleted but their injection priority drops over time.
  2. Source Reputation — auto-extracted memories begin in quarantine, while manually stored memories are not subject to the same low-confidence entry path. Memories from verified interactions score higher.
  3. Embedding Coherence — memories whose embeddings are consistent with the surrounding knowledge graph score higher. Outliers (potential hallucinations) score lower.
  4. Access Frequency — memories that are frequently recalled and not contradicted accumulate trust. This is a proxy for correctness: useful memories get accessed, wrong ones get forgotten or corrected.

The Quarantine System

Auto-extracted facts don't enter the active memory pool immediately. They start with a confidence score of 0.2 (quarantine zone). Only memories with confidence ≥ 0.3 AND a passing trust score are eligible for injection into agent context.

This prevents a critical failure mode: an LLM hallucinates a "fact" during extraction, and that hallucination gets injected into future sessions, causing a self-reinforcing error loop. The quarantine system breaks this loop by requiring memories to earn trust before influencing agents.

Memory Lifecycle: Consolidation and Decay

Memories aren't static. Over time, the memory store accumulates duplicates and outdated facts. memoryOSS handles this with two mechanisms:

  • Consolidation — similar memories are automatically detected and merged into a single, higher-confidence fact. This reduces noise and strengthens signal.
  • Decay — age-based cleanup gradually reduces the trust score of old memories. Facts that are never re-accessed eventually drop below the injection threshold and stop influencing agents, without being permanently deleted.

6Automatic Fact Extraction

When a request flows through the proxy, memoryOSS captures the LLM's response and extracts factual information asynchronously. This happens in the background — the response is returned to the client immediately, extraction doesn't add latency. Background extraction is only enabled automatically when a real provider API credential is configured; OAuth-only passthrough remains proxy-safe for traffic, but is not treated as a reliable extraction credential.

What Gets Extracted

  • Decisions — "We decided to use PostgreSQL instead of MongoDB"
  • Facts — "The UserService class handles authentication"
  • Preferences — "Use snake_case for all Python functions"
  • Problems and Solutions — "The timeout was caused by missing connection pooling"

What Doesn't Get Extracted

  • Conversational filler ("Sure, I can help with that")
  • Speculative content ("This might work...")
  • Secrets and credentials — filtered before extraction via pattern matching

Extraction Pipeline

Extraction uses the same upstream LLM (via a focused extraction prompt) and runs through three stages:

  1. Secret Redaction — API keys, tokens, passwords, and credentials are stripped before sending to extraction
  2. Fact Extraction — LLM extracts structured facts with confidence scores
  3. Embedding + Indexing — facts are embedded, indexed across all channels, and stored with initial quarantine confidence

Injection Filtering (NFKC-normalized)

Stored memories are filtered against prompt injection patterns before being injected into context. Unicode is NFKC-normalized to prevent homoglyph-based injection attacks (e.g., using Cyrillic characters that look like Latin to bypass filters).

7Security Model

memoryOSS sits in a sensitive position: between the client and the LLM API, with access to all requests and responses. The security model reflects this responsibility.

Authentication

  • All proxy endpoints require authenticated requests. In the default local setup this can be a generated memoryOSS proxy key or loopback-only passthrough of the client's own API key or OAuth token.
  • Keys are compared using constant-time SHA-256 hash comparison to prevent timing attacks
  • JWT tokens for session management, with separate signing keys for different purposes (auth vs. deletion certificates)
  • Optional IP allowlisting using actual connection source (not spoofable headers)

Namespace Isolation

Each API key maps to a namespace. Memories in namespace A are completely invisible to namespace B. This enables multi-tenant operation: different projects, teams, or users can share one memoryOSS instance without any data leakage.

Namespace keys are derived independently using AES-256-GCM encryption with per-namespace key derivation (Extract+Expand pattern). Path traversal between namespaces is prevented at the storage layer.

Data Protection

  • GDPR Export — /v1/export returns all stored data for a namespace
  • Certified Deletion — /v1/forget/certified with cryptographic deletion certificates
  • Memory Decay — age-based automatic cleanup
  • No telemetry by default — memoryOSS itself sends no analytics, crash reports, or phone-home traffic. Normal model requests still go to the upstream provider you configure.

Request Protection

  • Request body limit: 2 MB
  • Response body limit: 10 MB
  • Rate limiting with automatic stale bucket eviction
  • NFKC-normalized injection filtering on all stored content

8Design Decisions

Why Embedded Storage (redb)?

No PostgreSQL. No Redis. No Docker. memoryOSS uses redb, an embedded crash-safe key-value store. The entire database is a single file. This eliminates infrastructure complexity: no database server to configure, no connections to manage, no separate backup strategy. memoryoss backup produces a single compressed archive.

Why Local Embeddings (fastembed)?

Embedding generation runs locally using AllMiniLM-L6-V2 (384 dimensions). No OpenAI embedding API calls, no network latency, no per-token cost for embeddings. The model is small (~80MB) and fast enough for real-time recall. The embedding layer therefore works offline; full air-gapped operation requires a local upstream model instead of Anthropic/OpenAI.

Why Rust?

Memory safety without garbage collection pauses. Typical recall latency is 20–50ms, dominated by local embedding inference (AllMiniLM-L6-V2 on CPU). With persistent connections, overhead drops to single-digit milliseconds for cached queries. Single static binary with no runtime dependencies. Cross-compilation is straightforward.

Why Not a Vector Database?

Dedicated vector databases (Pinecone, Weaviate, Qdrant) are designed for large-scale similarity search across millions of vectors. Agent memory is typically thousands to tens of thousands of memories. At this scale, usearch (in-process, memory-mapped) outperforms any networked solution and eliminates an entire infrastructure dependency.

Why a Separate BM25 Index?

Embeddings compress information. The identifier handleWebSocketConnection and "manage network socket" produce similar vectors. But when a developer asks about handleWebSocketConnection, they mean that exact function, not anything semantically similar. BM25 full-text search preserves these distinctions. The dual-index approach (vector + BM25) is more reliable than either approach alone in identifier-heavy developer contexts.

10Conclusion

The AI agent memory problem is fundamentally an infrastructure problem, not an application problem. Asking every application to integrate memory SDKs doesn't scale. The proxy pattern solves this at the right layer: transparently, universally, and without requiring cooperation from application developers.

memoryOSS combines this architectural insight with multi-signal retrieval and injection control, safety mechanisms (quarantine, trust scoring, injection filtering), and operational simplicity (single binary, embedded storage, zero dependencies).

In our benchmark workload, a small set of early high-signal memories was inserted before a much larger background corpus. Subsequent queries continued to retrieve those early signal memories with 100% hit rate, while unrelated noise queries stayed below the injection threshold. Separate long-memory regression runs stored a sentinel first, added 5k or 20k background memories, and still recovered that first sentinel at rank 1. A separate 2-hour soak run completed without errors or leak signals. The public tests page now publishes the corresponding extraction-quality, observed-duplicate-rate, active-memory-size, and observed false-positive injection metrics alongside those retention artifacts. In a separate constrained 10-task Claude benchmark, replacing repeated full task context with recalled memory context reduced average input tokens by 44.4%. This is benchmark evidence for defined workloads, not a claim of perfect recall or universal token savings for every real-world conversation.

The result is a system where AI agents retain project-specific knowledge over time without giving every extracted fact equal weight.

memoryOSS is designed to preserve the context that makes work cumulative.
